# Pull Request Automation

!!! abstract "Overview"
    For each Security Task with a validated fix, TIVI automatically creates a pull request with full context — making it as easy as possible for developers to merge and close the finding.

## PR Structure

TIVI creates PRs that contain:

- The minimal code fix (validated in sandbox)
- A clear description explaining the security outcome
- The exploit evidence (sanitized — no live payloads in PR)
- The compliance requirements this satisfies
- A link to the full Exploit Report for deep-dive context

## PR Template

```markdown
## Security Fix: [Task Title]

### What This Fixes
[Plain-language description of the vulnerability and what an attacker could do]

### Why This Matters
[Business context — breach correlation, compliance requirements]

### Evidence
This vulnerability was confirmed exploitable in an isolated sandbox environment.
Exploit Report: [link to internal report]
Execution ID: [audit trail reference]

### What Changed
[Minimal description of the code change — 1-3 sentences]

### Compliance
This fix satisfies:
- NIST SSDF: [requirement IDs]
- SLSA: [requirements]  
- OpenSSF: [checks]

### Testing
- [x] Application starts successfully after patch
- [x] Existing test suite passes
- [x] Exploit payload confirmed blocked
- [x] No functional regression detected

---
*Generated by Threat-Informed Vulnerability Intelligence Platform*
*Validation timestamp: YYYY-MM-DD HH:MM:SS UTC*
*Audit trail: [execution-id]*
```

## Responsible Disclosure Integration

For vulnerabilities in third-party open-source packages (not your own code), TIVI follows responsible disclosure practices:

```yaml
# config/disclosure_policy.yaml
disclosure:
  initial_notification:
    method: private_security_contact   # security.txt, GitHub private advisory
    wait_days: 90                      # Standard coordinated disclosure window
    
  escalation:
    if_no_response_days: 30
    action: github_issue               # Public issue with limited details
    
  public_disclosure:
    after_days: 90
    or_after: patch_released
    action: github_pr                  # PR with full fix
    
  exceptions:
    critical_actively_exploited:
      wait_days: 7                     # Shorter window for KEV-listed vulnerabilities
```
